# 工作流：高可靠芯片设计JSON生成与最小代价修复 (workflow_generate_json.md)

## 1. 核心原则
- **绝对信任本地校验器**：`check_json_spec.py` 是最终的金标准（Oracle）。任何未通过校验的JSON都不可提交。
- **最小代价修正**：**严禁**因局部错误而全量重新生成整个芯片顶层JSON。采用**增量修补**与**模块级回滚**策略。
- **上下文解耦**：芯片设计具有天然的层次化（Top -> Sub-modules -> Instances/Ports）。利用这一特性拆分生成任务，降低单次LLM生成的上下文复杂度（Token数）。

---

## 2. 工作流阶段划分

### 阶段一：生成前（预防性架构设计）—— 规避结构性错误
*目标：将长文本生成转化为短文本并行生成，从根源上减少长序列的Attention分散与截断风险。*

1.  **生成“骨架（Skeleton）”而非“全量”**：
    - 先让LLM仅生成顶层模块的 **结构定义**（仅包含子模块实例名、类型、总线接口列表），不包含内部逻辑细节（RTL参数、内部连线）。此部分数据量小，准确率高。
2.  **拆分子模块（分治策略）**：
    - 针对每个子模块，单独发起LLM生成请求。指定上下文：“你正在生成芯片顶层 `Top` 下的子模块 `UART_CTRL`，其外部接口约束为 `{Skeleton_ports}`”。
    - 独立生成每个子模块的完整JSON。
3.  **预定义强制Schema**：
    - 在System Prompt中注入严格的 **JSON Schema**（使用 `description` 约束枚举值，如 `"type": "module", "sub_type": ["FSM", "DATAPATH", "MEMORY"]`），并提示LLM使用 `json.dumps` 思维（强制闭合括号）。

### 阶段二：初次组装与全量校验
4.  **本地拼接**：在本地（非LLM）将骨架与所有子模块JSON进行深度合并（Merge）。
5.  **运行校验**：执行 `python check_json_spec.py merged_design.json`。
    - **若通过**：结束流程。
    - **若失败**：进入阶段三（注意：此时必须捕获并保存 `check_json_spec.py` 的**完整标准错误输出 stderr**）。

---

### 阶段三：错误诊断与定位（最小化修复成本的关键）
*前提：`check_json_spec.py` 必须输出结构化的错误信息（建议脚本支持 `--json-errors` 参数）。若只输出文本，需用正则解析出行号或键值路径（JSON Path）。*

6.  **错误分类**（根据校验器报错）：
    - **类别A（全局键值错误）**：如顶层 `version` 缺失或 `clock_domain` 不匹配。
    - **类别B（子模块独立错误）**：如 `submodule_A` 内部寄存器地址冲突、参数越界。
    - **类别C（跨模块连线错误）**：如 `inst_UART` 连接的 `signal` 在顶层未声明。

### 阶段四：外科手术式定向修复（增量修补）
*策略：仅将错误的子模块上下文发给LLM，冻结未出错的部分。*

7.  **提取最小修复单元**：
    - 若错误属于**类别B**，从报错中提取出错的 **子模块ID**（如 `top.chip_top.inst_ram`）。
    - 若错误属于**类别A**，仅提取顶层缺失/错误的 **Key-Value**。
8.  **构建修复Prompt（关键）**：
    - **绝不**发送整个30MB的大JSON。
    - 发送内容：
        1. 该子模块的**父级接口约束**（仅上下行端口名）。
        2. 该子模块**当前的错误JSON片段**（出错的局部）。
        3. 校验器给出的**具体错误日志**（例如："ERROR: Submodule 'ram' size mismatch, expected 4096 got 2048"）。
        4. 指令：“请只返回修复后的 **该子模块（或该顶层字段）** 的完整JSON，不要包含其他模块。”
9.  **执行修复**：使用较低的 `temperature`（如 0.1）调用LLM，仅获取修复片段。

### 阶段五：替换、重测与回滚策略
10. **原地替换**：利用 `jq` 或 Python 脚本将修复后的片段覆盖回原合并文件中的对应节点（避免文本拼接引起的括号错乱）。
11. **增量校验**：再次运行 `check_json_spec.py`（可添加 `--only-check-modified` 加速，若无则全量跑）。
12. **回滚机制（最小代价保险）**：
    - 若同一子模块修复 **连续失败超过3次**，放弃自动修复，将该子模块**置为“黑盒（Black Box）”状态**（在JSON中标记 `"implementation": "locked"`），仅保留其接口定义，其余逻辑置空。此举确保顶层结构通过校验，代价仅为丢失该模块内部细节，而非整体设计作废。

---

## 3. 防御性编程与上下文管理（附加规则）

- **绝对长度警戒线**：在生成Prompt中要求LLM分块输出，若单次回复超过 **4096 tokens**，强制要求LLM使用紧凑格式（去除多余空格和换行，使用单行缩进）。
- **校验器增强建议**：既然你“绝对相信”该脚本，建议在脚本中增加 `--emit-patch` 功能，使其能输出标准化的 `jsonpatch` (RFC 6902) 操作指令，这样阶段四可直接应用Patch，无需LLM重新生成完整片段，进一步降低错误率。

---

## 4. 总结流程图（文本版）

```text
[开始] -> [1. 生成骨架] -> [2. 并行生成各子模块] -> [本地合并]
        -> [3. 运行 check_json_spec.py]
        -> [通过] -> [结束]
        -> [未通过] -> [解析stderr，定位模块路径]
        -> [4. 判断错误类型]
        -> [全局错误] -> [仅提取错误字段，发起修复请求]
        -> [模块错误] -> [提取该子模块上下文 + 错误日志，发起修复请求]
        -> [5. 本地应用Patch/替换]
        -> [6. 重跑校验]
        -> [重试次数 < 3] -> [循环修复]
        -> [重试次数 >= 3] -> [将该模块置为黑盒占位，强制通过顶层校验] -> [告警并结束]
```

此工作流通过 **分层生成** 消除长文本幻觉，通过 **局部定向修复** 将调试代价从 O(n) 降低为 O(1)，确保芯片设计JSON的交付可靠性。
